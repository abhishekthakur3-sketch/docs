## State Colors

Interactive elements transition through multiple states. Each state must remain visually distinguishable and meet contrast requirements.

| State | Behavior | Surface Token Pattern | Text/Icon Pattern |
|-------|----------|----------------------|-------------------|
| Default | Resting state | `surface.*.default` | `text.*.primary` |
| Hover | Cursor enters element | `surface.*.strongest` | `text.*.primary` |
| Focus | Keyboard navigation | `surface.*.default` + `border.info.primary` ring | `text.*.primary` |
| Active / Pressed | Click or tap | `surface.*.stronger` | `text.*.primary` |
| Disabled | Non-interactive | `surface.*.weakest` | `text.*.disabled` |
| Selected | Active selection | `surface.accent.default` | `text.heading.dark.primary` (on accent bg) |

<Info>Focus states should always include a visible border or ring using `border.info.primary` to ensure keyboard accessibility. Never rely solely on background color change for focus indication.</Info>

## Accessibility

<AccordionGroup>
  <Accordion title="Contrast Requirements (WCAG 2.1)" icon="eye">
    | Element | Minimum Ratio | Standard |
    |---------|--------------|----------|
    | Body text (< 18px) | 4.5:1 | AA |
    | Large text (≥ 18px or ≥ 14px bold) | 3:1 | AA |
    | UI components & graphical objects | 3:1 | AA |
    | Enhanced body text | 7:1 | AAA |
    | Enhanced large text | 4.5:1 | AAA |

    All TARMAC semantic tokens are designed to meet AA standards at minimum.
  </Accordion>
  <Accordion title="Color Blindness Considerations" icon="eye-low-vision">
    - Never rely on color alone to convey information (e.g., red = error). Always pair with icons, text labels, or patterns.
    - Error and success states must not depend solely on red/green differentiation — use icons (✓ / ✗) and text labels.
    - Test designs with color blindness simulators (Deuteranopia, Protanopia, Tritanopia).
  </Accordion>
  <Accordion title="Do / Don't" icon="list-check">
    **Do:**
    - Use `text.error.primary` on `surface.error.weakest` — high contrast, clear meaning
    - Add an error icon next to red error text
    - Test all color pairs with WebAIM Contrast Checker

    **Don't:**
    - Use `text.caption.disabled` on `surface.background.subtle` — insufficient contrast
    - Use only a red border to indicate an error field
    - Use alpha overlays below 20% for any text-bearing surface
  </Accordion>
</AccordionGroup>

## Edge Cases

| Scenario | Guidance |
|----------|----------|
| Disabled elements | Use `*.disabled` tokens. Reduce opacity is not recommended — use dedicated disabled color values to maintain contrast predictability. |
| Overlays & Modals | Use Alpha Black at 40% or 60% for scrim overlays. Content behind the overlay should be visually suppressed but not invisible. |
| Dense data UI (tables, lists) | Use `surface.background.strongest` for alternating row stripes. Avoid saturated colors for large data surfaces — keep them neutral. |
| Color on color | When placing colored text on a colored surface (e.g., white text on accent button), always verify the specific pair meets 4.5:1 contrast. |
| Skeleton / Loading states | Use `surface.neutral.weaker` for skeleton placeholders with a subtle shimmer animation. |
| Empty states | Use `surface.background.stronger` for illustration backgrounds, `text.caption.light.primary` for messaging. |

## Theme Strategy

TARMAC supports light and dark themes through semantic token remapping. The architecture ensures zero UI logic changes when switching themes.

<Tabs>
  <Tab title="How It Works">
    - Core colors remain fixed across themes — `Red.500` is always `#ED1B36`
    - Semantic tokens remap to different core values per theme
    - In light mode: `surface.background.default` → `White.500`
    - In dark mode: `surface.background.default` → `Black.500`
    - Components reference only semantic tokens, so they adapt automatically
  </Tab>
  <Tab title="Implementation">
    ```css
    /* Light theme (default) */
    :root {
      --surface-background-default: var(--color-white-500);
      --text-heading-primary: var(--color-black-200);
      --border-neutral-primary: var(--color-white-300);
    }

    /* Dark theme */
    [data-theme="dark"] {
      --surface-background-default: var(--color-black-500);
      --text-heading-primary: var(--color-white-450);
      --border-neutral-primary: var(--color-black-200);
    }
    ```
  </Tab>
  <Tab title="Future Extensions">
    The three-layer architecture supports:
    - **Brand themes**: Swap core palette for a partner brand while keeping semantic structure
    - **High contrast mode**: Remap semantic tokens to higher-contrast core values
    - **Seasonal / campaign themes**: Override accent tokens at the semantic layer
    - **Per-product theming**: Different products can share the system with custom accent mappings
  </Tab>
</Tabs>

## Validation & Testing

| Tool | Purpose | When to Use |
|------|---------|-------------|
| WebAIM Contrast Checker | Verify text/background contrast ratios | Every new color pairing |
| Chrome DevTools Accessibility Panel | Inspect computed contrast in browser | During development |
| Stark (Figma plugin) | Check contrast and simulate color blindness | During design |
| Color Oracle | System-wide color blindness simulation | Design review |
| axe DevTools | Automated accessibility audit | CI/CD pipeline |

<Steps>
  <Step title="Design Phase">
    Use Stark in Figma to verify all text/surface pairs meet AA contrast. Simulate Deuteranopia and Protanopia to ensure color-blind safety.
  </Step>
  <Step title="Development Phase">
    Use Chrome DevTools Accessibility panel to inspect rendered contrast. Run axe DevTools audit on each page.
  </Step>
  <Step title="QA Phase">
    Verify both light and dark themes. Check disabled states, error states, and dense data views. Use Color Oracle for full-screen color blindness simulation.
  </Step>
  <Step title="Ongoing">
    Add contrast checks to CI/CD. Flag any hardcoded hex values in code review. Audit new components against the token mapping table.
  </Step>
</Steps>

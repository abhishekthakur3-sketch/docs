## Overview

Color in the Tarmac Design System is built around a small set of core colors, extended into semantic tokens for surfaces, text, icons, and borders. The system separates raw values from semantic intent across three layers: Core → Semantic → Usage.

The color system serves as the visual language of TARMAC — it communicates brand identity, establishes hierarchy, signals interactivity, and ensures accessibility across every product surface. It is the single source of truth for all color decisions, shared between design (Figma) and engineering (code tokens).

## Principles

<AccordionGroup>
  <Accordion title="Semantic-First Usage" icon="tag">
    Always reference colors by their semantic role (`surface.error`, `text.heading.primary`), never by their raw hue (`crimson-500`). This ensures meaning is preserved even if underlying values change.
  </Accordion>
  <Accordion title="Consistency Through Constraints" icon="lock">
    The palette is intentionally constrained. Every color in the system has a defined purpose. Avoid introducing ad-hoc colors — if a new need arises, extend the system at the token layer.
  </Accordion>
  <Accordion title="Accessibility by Default" icon="universal-access">
    Contrast ratios are enforced at the semantic layer. Body text meets WCAG AA (4.5:1), large text meets 3:1, and UI elements meet 3:1 against their backgrounds. Accessibility is structural, not manual.
  </Accordion>
  <Accordion title="Scalability" icon="arrows-up-down-left-right">
    Adding a new brand, theme, or product variant should only require changes at the core or semantic layer — never at the component level. The architecture supports light/dark modes and future brand extensions.
  </Accordion>
  <Accordion title="Neutral First" icon="circle">
    Neutrals form the foundation of every interface. Accent and status colors are used sparingly and purposefully for emphasis, feedback, and wayfinding.
  </Accordion>
</AccordionGroup>

## Color Architecture

The system is organized in three layers. Each layer builds on the one below it, creating a clear separation between raw values and their application in UI.

<Steps>
  <Step title="Layer 1: Core Colors">
    Fixed, raw color values representing the full spectrum. These are the palette — they define what colors exist. Core colors are never referenced directly in UI.
  </Step>
  <Step title="Layer 2: Semantic Tokens">
    Purpose-driven mappings that assign meaning to core colors. Tokens like `surface.error.default` or `text.heading.primary` describe intent, not hue. If a color changes, the semantic name stays stable.
  </Step>
  <Step title="Layer 3: Usage (Components)">
    Components consume semantic tokens. A Button uses `surface.accent.default` for its background and `text.heading.light.primary` for its label. Components never know about raw hex values.
  </Step>
</Steps>

<Warning>Core colors are palette-only. No component or screen should reference them directly. All UI usage must go through semantic tokens.</Warning>

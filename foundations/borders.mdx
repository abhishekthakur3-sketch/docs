## Overview

Borders help define structure, separation, and hierarchy across the interface. They guide the eye, reinforce component states, and provide clarity without overwhelming the layout. In this system, borders are intentionally minimal — used to support content rather than compete with it.

The border scale includes stroke widths from 0.5px to 4px, with semantic types (success, error, warning, link) and states (default, hovered, pressed, disabled) for consistent meaning across components.

## Stroke Tokens

| Token | Value | Purpose |
|-------|-------|---------|
| `stroke.small` | 0.5px | Ultra-subtle separators, table grids, low-emphasis dividers |
| `stroke.default` | 1px | Default border for UI components and containers |
| `stroke.medium` | 1.5px | High-emphasis interactive states, active boundaries, focus outlines |
| `stroke.large` | 2px | Strong structural separation, important emphasis elements |
| `stroke.xlarge` | 4px | Rare use — hero, overlays, high-priority interaction or identity elements |

---

## Border States

| State | Behavior | Visual Treatment |
|-------|----------|-----------------|
| Default | Resting state | Neutral or semantic color at standard thickness |
| Hovered | Cursor enters element | Slightly stronger contrast and/or thicker stroke |
| Pressed | Click or tap | Highest-emphasis state border |
| Disabled | Non-interactive | Reduced contrast, minimal visual presence |

---

## Semantic Border Types

Four semantic border types, each applied consistently across components:

| Type | Purpose | Example Usage |
|------|---------|---------------|
| Success | Positive confirmation | Validated inputs, success alerts, confirmed selections |
| Error | Critical validation or failure | Error inputs, failed submissions, destructive actions |
| Warning | Cautionary or attention-required | Warning alerts, expiring items, limit-approaching states |
| Link / Info | Informative or navigational emphasis | Focus rings, info callouts, active navigation |

Each semantic type adapts across light and dark modes, with tonal steps for disabled → primary → secondary → tertiary usage.

## Usage by Context

| Context | Examples | Token | Value |
|---------|----------|-------|-------|
| Subtle separators | Table grids, list dividers, card edges | `stroke.small` | 0.5px |
| Standard components | Inputs, buttons, cards, dropdowns | `stroke.default` | 1px |
| Interactive emphasis | Focus rings, active tabs, selected items | `stroke.medium` | 1.5px |
| Structural separation | Section dividers, modal frames | `stroke.large` | 2px |
| High-priority elements | Hero sections, identity framing | `stroke.xlarge` | 4px |

## Principles

<AccordionGroup>
  <Accordion title="Clarity Over Decoration">
    Borders exist to guide the eye — not to add style or noise.
  </Accordion>
  <Accordion title="Light First">
    Thin strokes are preferred to maintain a clean, balanced visual tone.
  </Accordion>
  <Accordion title="Hierarchy Through Weight">
    Thicker borders imply stronger separation or interaction significance.
  </Accordion>
  <Accordion title="Consistency Across Surfaces">
    The same border weight should communicate the same purpose everywhere.
  </Accordion>
  <Accordion title="Integrate With Radius and Spacing">
    Borders reinforce — not disrupt — the structural system.
  </Accordion>
</AccordionGroup>

## Accessibility

Borders improve usability by:

- Reinforcing visual grouping and separating layers
- Improving contrast where shadows are insufficient
- Supporting scannability in dense layouts
- Guiding focus across workflows and interactive elements

Thin but clear borders reduce cognitive load — especially in complex operational UI — ensuring users never have to guess structure.

## Usage Guidelines

<CardGroup cols={2}>
  <Card title="✓ Do" icon="check" color="#16A34A">
    - Use semantic stroke tokens instead of raw pixel values
    - Match stroke weight to element importance
    - Pair borders with radius tokens for consistent geometry
    - Use focus borders (`stroke.medium`) for keyboard navigation
  </Card>
  <Card title="✗ Don't" icon="xmark" color="#DC2626">
    - Don't use 4px borders on standard UI components
    - Don't mix border weights within the same component
    - Don't rely solely on borders for state communication
    - Don't use borders where spacing alone provides separation
  </Card>
</CardGroup>

## Developer Handoff

<CodeGroup>
```css CSS Variables
/* Border stroke tokens */
--stroke-small: 0.5px;
--stroke-default: 1px;
--stroke-medium: 1.5px;
--stroke-large: 2px;
--stroke-xlarge: 4px;
```

```tsx TypeScript
export const stroke = {
  small: '0.5px',
  default: '1px',
  medium: '1.5px',
  large: '2px',
  xlarge: '4px',
} as const;
```

```tsx Usage
// Input with default border
<Input style={{ border: 'var(--stroke-default) solid var(--border-neutral-primary)' }} />

// Focus ring
<Input style={{ border: 'var(--stroke-medium) solid var(--border-info-primary)' }} />

// Card with subtle border
<Card style={{ border: 'var(--stroke-small) solid var(--border-neutral-secondary)' }} />
```

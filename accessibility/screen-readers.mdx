## Overview

TARMAC components are built with semantic HTML and proper ARIA attributes to ensure screen readers can accurately convey content, structure, and interactive states to users.

---

## Semantic HTML First

Always use the correct HTML element. Screen readers rely on element semantics to communicate purpose:

| Instead of | Use |
|------------|-----|
| `div` with click handler | `button` |
| `div` with href behavior | `a` with `href` |
| `div` for navigation | `nav` with `aria-label` |
| `div` for main content | `main` |
| `span` for heading | `h1` through `h6` |
| `div` for list | `ul`/`ol` with `li` |

## ARIA Attributes

### When to Use ARIA

<Info>
  The first rule of ARIA: don't use ARIA if native HTML can do the job. ARIA supplements HTML — it doesn't replace it.
</Info>

### Common ARIA Attributes in TARMAC

| Attribute | Purpose | Example |
|-----------|---------|---------|
| `aria-label` | Accessible name when no visible text | Icon buttons, search inputs |
| `aria-labelledby` | References a visible labeling element | Dialog titles, section headings |
| `aria-describedby` | References additional description | Helper text, error messages |
| `aria-hidden="true"` | Hides decorative elements | Decorative icons, dividers |
| `aria-live` | Announces dynamic content changes | Toast messages, status updates |
| `aria-expanded` | Indicates expandable state | Accordions, dropdowns |
| `aria-selected` | Indicates selection state | Tabs, list items |
| `aria-invalid` | Indicates validation error | Form inputs with errors |
| `aria-required` | Indicates required field | Mandatory form inputs |

## Live Regions

Use live regions to announce dynamic content changes without moving focus:

| Type | Attribute | Behavior | Use For |
|------|-----------|----------|---------|
| Polite | `aria-live="polite"` | Waits for user pause | Status updates, confirmations |
| Assertive | `aria-live="assertive"` | Interrupts immediately | Error messages, urgent alerts |
| Status | `role="status"` | Polite by default | Loading states, progress |
| Alert | `role="alert"` | Assertive by default | Validation errors, warnings |

```tsx
// Polite: status updates
<div aria-live="polite" aria-atomic="true">
  {statusMessage}
</div>

// Assertive: error alerts
<div role="alert">
  {errorMessage}
</div>
```

## Page Landmarks

Define clear page regions so screen reader users can jump between sections:

| Landmark | HTML Element | Purpose |
|----------|-------------|---------|
| Banner | `header` | Site header |
| Navigation | `nav` | Navigation menus |
| Main | `main` | Primary content |
| Complementary | `aside` | Sidebar content |
| Content Info | `footer` | Site footer |
| Search | `role="search"` | Search functionality |

<Tip>
  When using multiple nav elements, give each a unique aria-label (e.g., "Main navigation", "Footer navigation") so screen reader users can distinguish them.
</Tip>

## Heading Hierarchy

Maintain a logical heading structure:

| Rule | Details |
|------|---------|
| One H1 per page | Describes the page purpose |
| Sequential order | Never skip levels (H1 then H3) |
| Descriptive text | Headings should summarize the section |
| Don't use for styling | Use CSS for visual size, headings for structure |

## Testing Checklist

| Check | Pass Criteria |
|-------|---------------|
| All content announced | Screen reader reads all visible text |
| Interactive elements labeled | Buttons, links, inputs have accessible names |
| Form fields associated | Labels linked to inputs via htmlFor/id |
| Errors announced | Validation errors read aloud when they appear |
| Dynamic updates announced | Toasts and status changes use live regions |
| Landmarks present | Page has header, nav, main, footer landmarks |
| Heading hierarchy logical | H1 through H6 in correct order |
| Decorative elements hidden | Icons and dividers use aria-hidden |

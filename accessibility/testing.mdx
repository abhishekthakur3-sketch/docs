## Overview

Comprehensive accessibility testing combines automated tools, manual checks, and real user testing. No single method catches everything — use all three for full coverage.

<CardGroup cols={3}>
  <Card title="Automated" icon="robot">
    Catches ~30% of issues: missing alt text, contrast, ARIA errors
  </Card>
  <Card title="Manual" icon="hand">
    Catches ~50% of issues: keyboard flow, focus order, screen reader experience
  </Card>
  <Card title="User Testing" icon="users">
    Catches ~20% of issues: real-world usability with assistive tech
  </Card>
</CardGroup>

## Automated Testing

### Browser Extensions

| Tool | What It Catches |
|------|----------------|
| axe DevTools | ARIA errors, contrast, missing labels, structure |
| WAVE | Visual overlay of issues, contrast, landmarks |
| Lighthouse | Accessibility score with actionable recommendations |

### Code-Level Testing

Add accessibility checks to your test suite:

```tsx
import { axe, toHaveNoViolations } from 'jest-axe';

expect.extend(toHaveNoViolations);

test('component has no accessibility violations', async () => {
  const { container } = render(<MyComponent />);
  const results = await axe(container);
  expect(results).toHaveNoViolations();
});
```

### CI/CD Integration

Run accessibility tests on every pull request:

```yaml
# .github/workflows/a11y.yml
name: Accessibility
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run test:a11y
```

## Manual Testing

### Keyboard Testing

| Test | How | Pass Criteria |
|------|-----|---------------|
| Tab navigation | Press Tab through entire page | All interactive elements reachable |
| Focus visibility | Tab and observe | Clear focus ring on every element |
| Focus order | Tab and check sequence | Matches visual layout |
| Activation | Press Enter/Space on controls | Buttons, links, toggles respond |
| Escape | Press Escape on overlays | Modals and dropdowns close |
| Arrow keys | Navigate tabs, menus, radios | Composite widgets respond |
| Skip link | Tab once from page top | Skip link appears and works |

### Screen Reader Testing

| Test | How | Pass Criteria |
|------|-----|---------------|
| Content reading | Navigate with screen reader | All visible text announced |
| Interactive labels | Focus buttons, links, inputs | Clear accessible names announced |
| Form fields | Navigate form | Labels, required state, errors announced |
| Dynamic content | Trigger toasts, status changes | Updates announced via live regions |
| Landmarks | Use landmark navigation | Header, nav, main, footer present |
| Headings | Use heading navigation | Logical H1-H6 hierarchy |

### Visual Testing

| Test | How | Pass Criteria |
|------|-----|---------------|
| Text contrast | Use contrast checker tool | 4.5:1 for normal text, 3:1 for large |
| Color blindness | Use Color Oracle or similar | Information not conveyed by color alone |
| Text scaling | Zoom browser to 200% | Content remains usable, no overflow |
| High contrast | Enable OS high contrast mode | All content visible and usable |
| Reduced motion | Enable prefers-reduced-motion | Animations disabled or reduced |

## When to Test

| Phase | What to Test |
|-------|-------------|
| Design | Color contrast, touch targets, visual hierarchy |
| Development | Keyboard navigation, ARIA attributes, semantic HTML |
| Code Review | Automated axe checks, focus management |
| QA | Full manual testing with screen readers |
| Release | Regression testing on key user flows |

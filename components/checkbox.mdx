# Checkbox

## Overview

Checkboxes are interactive selection controls that enable users to choose one or more items independently from a group of options. Unlike radio buttons, checkboxes allow multiple selections simultaneously. They are commonly used in forms, filters, preference settings, and bulk-selection interfaces. Checkboxes provide clear visual feedback when options are selected, unselected, or partially selected.

## When to Use

<AccordionGroup>
  <Accordion title="When to Use" icon="check">
    - Selecting multiple filters or categories
    - Managing feature preferences
    - Bulk selecting items in tables
    - Allowing users to toggle optional settings
    - Accepting terms or conditions
    - Enabling or disabling specific features
  </Accordion>

  <Accordion title="When Not to Use" icon="xmark">
    - When only one option can be selected (use radio buttons)
    - For triggering immediate actions (use buttons or toggles)
    - When the option requires immediate confirmation
    - For binary on/off states where a toggle switch is more appropriate
  </Accordion>
</AccordionGroup>

### Checkbox vs Radio Button vs Toggle

| Control | Selection | Behavior | Use Case |
|---------|-----------|----------|----------|
| **Checkbox** | Multiple | Independent toggle per item | Multi-select filters, preferences |
| **Radio Button** | Single | Mutually exclusive within group | Single choice from options |
| **Toggle** | Binary | Immediate on/off action | Feature switches, settings |

## Anatomy

The Checkbox component consists of the following elements:

- **Checkbox Control**: The square or rounded interactive element that indicates the selection state
- **Indicator**: Appears inside the checkbox when selected — displays a checkmark (checked) or a horizontal line (partial/indeterminate)
- **Label**: The text describing the option associated with the checkbox
- **Supporting Text** (optional): Additional information explaining the option when needed

<Info>Checkboxes should maintain consistent spacing between the control and label to ensure readability and usability.</Info>

## Styles

Styles determine the shape of the checkbox control.

### Box (Square)
Uses a square container with slightly rounded corners. Suitable for most interfaces and form elements.

```jsx
<Checkbox style="box" label="Accept terms" />
```

### Rounded
Uses a fully rounded container. Often used when the interface adopts a softer or more modern visual style.

```jsx
<Checkbox style="rounded" label="Accept terms" />
```

<Tip>Both styles communicate the same interaction behavior and are primarily a visual preference. Choose one and use it consistently across your product.</Tip>

## Sizes

Checkbox sizes adapt the component to different interface densities.

| Size | Dimension | Usage |
|------|-----------|-------|
| **Large** | 24 × 24px | Spacious layouts or forms where readability is important |
| **Medium** | 20 × 20px | Default size used in most interfaces |
| **Small** | 16 × 16px | Dense environments such as tables or compact lists |

<Tip>Use Medium (20px) as the default size for most interfaces. Even when smaller checkboxes are used, the clickable area should remain large enough (minimum 44×44px) for comfortable interaction.</Tip>

## States

Checkboxes visually respond to interaction states to communicate selection feedback.

### Default
Checkbox is unselected and inactive. Displays an empty container with a visible border.

### Hover
Indicates pointer interaction and highlights the control. Background or border color subtly changes to signal interactivity.

### Checked
Displays the checkmark indicator inside the container, confirming the option is selected. The container fills with the variant color.

### Partial (Indeterminate)
Indicates that some but not all nested options are selected. Displays a horizontal line inside the container. Often used in hierarchical selections such as table headers or parent-child checkbox groups.

### Disabled
Indicates the checkbox cannot be interacted with. Reduced opacity, no hover effects, and cursor shows not-allowed. Can appear in both checked and unchecked states.

<Info>State changes should provide clear visual feedback so users understand the current selection status.</Info>

## Interaction Behavior

### Click/Tap
- Clicking the checkbox toggles the selection state
- Clicking the label should also toggle the checkbox
- Provides immediate visual feedback

### Partial/Indeterminate Logic
- Partial states appear when only some items in a group are selected
- In hierarchical selections, selecting a parent checkbox selects all child options
- Deselecting a parent checkbox deselects all child options
- When some children are selected, the parent shows the partial indicator

### Disabled Behavior
- Disabled checkboxes remain visible but cannot be interacted with
- Disabled state can apply to both checked and unchecked checkboxes
- Not included in keyboard tab order

### Hover (Desktop)
- Border or background color changes subtly
- Cursor changes to pointer
- Smooth transition (150ms)

### Touch (Mobile)
- Minimum 44×44px touch target regardless of visual size
- Active state on touch down
- Action triggers on touch up

## Usage Guidelines

### Do's and Don'ts

<CardGroup cols={2}>
  <Card title="✓ Do" icon="check" color="#10b981">
    **Use clear descriptive labels**
    - "Receive email notifications"
    - "Accept terms and conditions"

    **Group related checkboxes**
    - Organize logically
    - Use clear group headings

    **Provide visible feedback**
    - Show check indicator on selection
    - Use partial states for hierarchical groups

    **Use partial states correctly**
    - For parent-child relationships
    - In table header selections
  </Card>

  <Card title="✗ Don't" icon="xmark" color="#ef4444">
    **Don't use for single selection**
    - Use radio buttons instead
    - Checkboxes imply multi-select

    **Don't hide important options**
    - Avoid long scrollable checkbox lists
    - Keep groups manageable (5-7 items)

    **Don't omit labels**
    - Every checkbox needs a label
    - Icon-only checkboxes need aria-label

    **Don't overcrowd layouts**
    - Maintain adequate spacing
    - Use clear visual grouping
  </Card>
</CardGroup>

### Best Practices

<Steps>
  <Step title="Use clear, descriptive labels">
    Labels should clearly describe what the checkbox controls. Avoid ambiguous text.
  </Step>
  <Step title="Group related options">
    Organize checkboxes into logical groups with clear headings when presenting multiple options.
  </Step>
  <Step title="Maintain consistent spacing">
    Use 8px minimum between checkboxes in a group, 16px between checkbox groups.
  </Step>
  <Step title="Place labels to the right">
    Labels should appear to the right of the checkbox control for left-to-right languages.
  </Step>
  <Step title="Use indeterminate state for hierarchies">
    When a parent checkbox controls child checkboxes, show the partial state when only some children are selected.
  </Step>
</Steps>

## Engineering Implementation

### Component API

```typescript
interface CheckboxProps {
  // Appearance
  variant?: 'standard' | 'blue' | 'green' | 'dlv-red';
  size?: 'sm' | 'md' | 'lg';
  style?: 'box' | 'rounded';

  // State
  checked?: boolean;
  indeterminate?: boolean;
  disabled?: boolean;
  defaultChecked?: boolean;

  // Content
  label?: string;
  supportingText?: string;

  // Behavior
  name?: string;
  value?: string;
  onChange?: (event: ChangeEvent<HTMLInputElement>) => void;

  // Styling
  className?: string;
  ariaLabel?: string;
  ariaDescribedBy?: string;
}
```

### Implementation Notes

**Technical Requirements:**
- Use native `<input type="checkbox">` for proper semantics
- Support controlled and uncontrolled modes
- Handle indeterminate state via ref (`inputRef.current.indeterminate = true`)
- Use CSS transitions for smooth state changes (150ms)
- Ensure minimum 44×44px touch target regardless of visual size
- Clicking the label must toggle the checkbox

**Variant Hierarchy:**
- Standard (Black) is the default for most interfaces
- Blue for primary action themes
- Green for positive/confirmation contexts
- DLV Red for brand-specific emphasis

```jsx Indeterminate (Select All)
function SelectAllExample() {
  const [items, setItems] = useState([false, true, false]);
  const allChecked = items.every(Boolean);
  const someChecked = items.some(Boolean);

  return (
    <>
      <Checkbox
        label="Select all"
        checked={allChecked}
        indeterminate={someChecked && !allChecked}
        onChange={() => setItems(items.map(() => !allChecked))}
      />
      {items.map((checked, i) => (
        <Checkbox
          key={i}
          label={`Item ${i + 1}`}
          checked={checked}
          onChange={() => {
            const next = [...items];
            next[i] = !next[i];
            setItems(next);
          }}
        />
      ))}
    </>
  );
}
```

## Design References

### Spacing and Alignment
- Checkbox control aligns to the top of the first line of label text
- 8px gap between checkbox control and label
- 8px minimum vertical spacing between checkboxes in a group
- 16px spacing between checkbox groups
- Labels placed to the right of the checkbox control

### Interaction Consistency
- All checkbox sizes use the same transition timing (150ms ease-in-out)
- Hover state applies to both the control and its label area
- Focus ring appears on keyboard navigation only (`:focus-visible`)
- Touch targets maintain 44×44px minimum regardless of visual checkbox size

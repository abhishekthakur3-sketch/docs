# Button

## Overview

Buttons are interactive elements that trigger actions when clicked or activated. They are one of the most fundamental components in the TARMAC Design System, used to submit forms, navigate between pages, open dialogs, and perform various user-initiated actions. Buttons should clearly communicate their purpose and provide appropriate visual feedback.

## Anatomy

The Button component consists of the following elements:

- **Container**: The clickable area with background and border
- **Label**: Text describing the button's action
- **Icon** (optional): Visual indicator supporting the label (left or right position)
- **Loading Indicator** (optional): Spinner or animation during async operations
- **Focus Ring**: Visual indicator for keyboard navigation

<Info>Icons should only be included when they add clarity to the button's purpose.</Info>

## Variants

### Primary
The main call-to-action button with high visual prominence. Use sparingly for the most important action on a page.

```jsx
<Button variant="primary">Save Changes</Button>
```

<Tip>Use only one primary button per section to maintain clear visual hierarchy.</Tip>

### Secondary
Supporting actions with medium visual weight. Used for secondary actions alongside primary buttons.

```jsx
<Button variant="secondary">Cancel</Button>
```

### Tertiary
Low-emphasis actions that don't require strong visual weight.

```jsx
<Button variant="tertiary">Learn More</Button>
```

### Destructive
Actions that delete, remove, or perform irreversible operations. Uses red/warning colors.

```jsx
<Button variant="destructive">Delete Account</Button>
```

<Warning>Always pair destructive actions with confirmation dialogs.</Warning>

### Ghost
Minimal styling for subtle actions, often used in toolbars or dense interfaces.

```jsx
<Button variant="ghost">Edit</Button>
```

### Link
Styled as a hyperlink but with button semantics and behavior.

```jsx
<Button variant="link">View Details</Button>
```

## Sizes

Buttons are available in three sizes to accommodate different UI contexts:

| Size | Height | Padding | Font Size | Usage |
|------|--------|---------|-----------|-------|
| **SM (Small)** | 32px | 12px horizontal | 14px | Compact spaces, toolbars, tables |
| **MD (Medium)** | 40px | 16px horizontal | 16px | Default size for most use cases |
| **LG (Large)** | 48px | 20px horizontal | 18px | Prominent actions, hero sections |

<Tip>Use MD (Medium) as the default size for most interfaces.</Tip>

## States

### Default
Standard appearance ready for interaction.

### Hover
Visual feedback when cursor is over the button.
- Background color darkens or lightens
- Subtle scale or shadow effect

### Active/Pressed
Visual feedback during click or tap.
- Slightly darker than hover state
- Reduced elevation or inset appearance

### Focus
Visible outline for keyboard navigation.
- 2px focus ring with high contrast
- Maintains accessibility standards

### Disabled
Non-interactive state indicating unavailable action.
- Reduced opacity (40%)
- No hover effects
- Cursor shows not-allowed
- Not keyboard focusable

### Loading
Indicates async operation in progress.
- Loading spinner replaces or accompanies label
- Button remains disabled during loading
- Maintains button dimensions

## Interaction Behavior

### Click/Tap
- Provides immediate visual feedback (active state)
- Triggers associated action or navigation
- Shows loading state for async operations
- Prevents double-submission during processing

### Hover (Desktop)
- Background color darkens slightly
- Cursor changes to pointer
- Smooth transition (150ms)
- No layout shift

### Focus (Keyboard)
- Visible focus ring appears (2px outline)
- Accessible via Tab key navigation
- Activated with Enter or Space key
- Focus ring uses high-contrast color

### Touch (Mobile)
- Minimum 44x44px touch target
- Active state on touch down
- Action triggers on touch up
- Haptic feedback (when supported)

## Usage Guidelines

<AccordionGroup>
  <Accordion title="When to Use" icon="check">
    - Submitting forms or saving data
    - Navigating to different pages or sections
    - Opening modals, dialogs, or menus
    - Confirming or canceling actions
    - Triggering any user-initiated action
  </Accordion>
  
  <Accordion title="When Not to Use" icon="xmark">
    - For navigation that should be a link (use anchor tags)
    - When a checkbox or toggle is more appropriate
    - For non-interactive labels or status indicators
    - When the action is not immediately available
  </Accordion>
</AccordionGroup>

### Do's and Don'ts

<CardGroup cols={2}>
  <Card title="✓ Do" icon="check" color="#10b981">
    **Use action verbs**
    - "Save Changes"
    - "Delete Account"
    - "Add to Cart"
    
    **Keep labels concise**
    - 1-3 words ideal
    - Clear and specific
    
    **Provide feedback**
    - Loading states
    - Success confirmation
    - Error handling
  </Card>
  
  <Card title="✗ Don't" icon="xmark" color="#ef4444">
    **Avoid vague labels**
    - ~~"Click Here"~~
    - ~~"Submit"~~
    - ~~"OK"~~
    
    **Don't overuse primary**
    - Only one per section
    - Maintain hierarchy
    
    **Avoid disabled without context**
    - Explain why disabled
    - Show requirements
  </Card>
</CardGroup>

### Best Practices

<Steps>
  <Step title="Use clear, action-oriented labels">
    Start with verbs: "Save", "Delete", "Create", "Cancel"
  </Step>
  <Step title="Maintain visual hierarchy">
    One primary button per section, secondary for supporting actions
  </Step>
  <Step title="Provide loading states">
    Show spinners or disable during async operations
  </Step>
  <Step title="Ensure adequate spacing">
    Minimum 8px between buttons, 16px for button groups
  </Step>
  <Step title="Consider mobile touch targets">
    Minimum 44x44px for touch-friendly interactions
  </Step>
</Steps>

## Accessibility

<Info>All button components are built to meet WCAG 2.1 Level AA standards.</Info>

### Semantic HTML
Always use the native `<button>` element for proper semantics and keyboard support.

```jsx
<button type="button">Click Me</button>
<button type="submit">Submit Form</button>
<button type="reset">Reset Form</button>
```

### ARIA Labels and States

<CodeGroup>
```jsx Icon-Only Button
<Button aria-label="Close dialog">
  <CloseIcon />
</Button>
```

```jsx Loading State
<Button aria-busy="true" disabled>
  <Spinner /> Loading...
</Button>
```

```jsx Pressed State (Toggle)
<Button aria-pressed={isActive}>
  {isActive ? 'Active' : 'Inactive'}
</Button>
```

```jsx Expanded State (Dropdown)
<Button aria-expanded={isOpen} aria-haspopup="menu">
  Menu <ChevronIcon />
</Button>
```
</CodeGroup>

### Accessibility Checklist

<AccordionGroup>
  <Accordion title="✓ Visual Requirements" icon="eye">
    - [ ] Text contrast ratio ≥ 4.5:1 (WCAG AA)
    - [ ] Focus indicator contrast ratio ≥ 3:1
    - [ ] Focus ring visible on all interactive states
    - [ ] Minimum 44x44px touch target size
    - [ ] No color-only information
  </Accordion>
  
  <Accordion title="✓ Keyboard Requirements" icon="keyboard">
    - [ ] Focusable via Tab key
    - [ ] Activatable with Enter and Space
    - [ ] Disabled buttons not in tab order
    - [ ] Focus order follows visual order
    - [ ] No keyboard traps
  </Accordion>
  
  <Accordion title="✓ Screen Reader Requirements" icon="audio-description">
    - [ ] Descriptive, action-oriented labels
    - [ ] Icon-only buttons have aria-label
    - [ ] Loading states announced with aria-busy
    - [ ] Disabled state communicated
    - [ ] Button role implicit or explicit
  </Accordion>
</AccordionGroup>

## Engineering Implementation

### Component API

```typescript
interface ButtonProps {
  // Appearance
  variant?: 'primary' | 'secondary' | 'tertiary' | 'destructive' | 'ghost' | 'link';
  size?: 'sm' | 'md' | 'lg';
  
  // States
  disabled?: boolean;
  loading?: boolean;
  
  // Layout
  fullWidth?: boolean;
  
  // Icon support
  icon?: ReactNode;
  iconPosition?: 'left' | 'right';
  
  // Behavior
  type?: 'button' | 'submit' | 'reset';
  onClick?: (event: MouseEvent) => void;
  
  // Content
  children: ReactNode;
  
  // Styling
  className?: string;
  ariaLabel?: string;
}
```

### Implementation Notes

**Technical Requirements:**
- Use native `<button>` element for proper semantics
- Prevent default form submission when type="button"
- Disable pointer events when loading or disabled
- Use CSS transitions for smooth state changes (150ms)
- Ensure minimum touch target size (44x44px on mobile)
- Handle loading state without layout shift

**Button Hierarchy:**
- Use only one primary button per section
- Secondary buttons support the primary action
- Tertiary/ghost buttons for less important actions
- Destructive buttons should be visually distinct

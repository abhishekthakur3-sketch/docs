## Overview

Input fields are fundamental form components used to collect user input — names, values, search queries, configuration parameters, and system inputs. The component provides contextual information through labels, helper text, status indicators, and validation messages. It supports three structural layers: Title Container, Input Field Container, and Subtext Container.

## Anatomy

The Input component consists of the following elements:

- **Container**: Wrapper that defines the input boundary
- **Input Field**: The interactive text entry area
- **Label**: Descriptive text identifying the input purpose
- **Placeholder**: Hint text shown when input is empty
- **Helper Text**: Additional guidance or instructions
- **Error Message**: Validation feedback for incorrect input
- **Prefix/Suffix**: Icons or text before/after the input value
- **Clear Button** (optional): Quick way to empty the input

<Info>Always provide clear, descriptive labels above inputs for better accessibility and mobile experience.</Info>

## Variants

### Text Input
Standard single-line text entry for names, emails, and general text.

```jsx
<Input type="text" label="Full Name" />
```

### Email Input
Specialized for email addresses with built-in validation.

```jsx
<Input type="email" label="Email Address" />
```

### Password Input
Masked text entry with optional visibility toggle.

```jsx
<Input type="password" label="Password" />
```

### Number Input
Numeric entry with optional increment/decrement controls.

```jsx
<Input type="number" label="Quantity" min={1} max={100} />
```

### Search Input
Optimized for search functionality with search icon and clear button.

```jsx
<Input type="search" placeholder="Search..." />
```

### Textarea
Multi-line text entry for longer content.

```jsx
<Textarea label="Description" rows={4} />
```

Input fields are available in three sizes:

| Size | Height | Padding | Font Size | Usage |
|------|--------|---------|-----------|-------|
| **SM (Small)** | 32px | 12px horizontal | 14px | Compact forms, filters, toolbars |

| **LG (Large)** | 48px | 12px horizontal | 18px | Prominent forms, landing pages |

<Tip>Use MD (Medium) as the default size for most form inputs.</Tip>

## States

### Default
Ready for user input with neutral styling.

### Hover
Subtle visual feedback when cursor is over the input.
- Border color slightly darkens
- Maintains clear interactive affordance

### Focus
Active state when user is typing or input has focus.
- Prominent border color (primary color)
- Visible focus ring for accessibility (2px outline)
- Helper text may appear or change

### Filled
Input contains user-entered value.
- Label may move or shrink (floating label pattern)
- Clear button may appear

### Disabled
Non-interactive state preventing user input.
- Reduced opacity (60%)
- Grayed out appearance
- No hover or focus effects
- Not keyboard focusable

### Error
Invalid input requiring user correction.
- Red border color
- Error message displayed below
- Error icon may appear
- Maintains focus capability

### Success
Valid input confirmation (optional).
- Green border or checkmark icon
- Success message may appear

### Read-Only
Displays value but prevents editing.
- Similar to disabled but maintains normal contrast
- Focusable and selectable
- No edit cursor

## Interaction Behavior

### Focus
- Border color changes to primary color
- Focus ring appears (2px outline)
- Label may animate (floating label pattern)
- Helper text may appear or update
- Cursor appears in input field

### Typing
- Characters appear as user types
- Real-time validation (optional)
- Character counter updates (if enabled)
- Autocomplete suggestions may appear

### Blur (Lose Focus)
- Validation triggers
- Error messages display if invalid
- Border returns to default state
- Value persists in field

### Clear Action
- Clear button appears when input has value
- Clicking clear removes all text
- Focus remains in input field
- Triggers onChange event

## Usage Guidelines

<AccordionGroup>
  <Accordion title="When to Use" icon="check">
    - Collecting user information in forms
    - Search functionality
    - Filtering and sorting data
    - Settings and configuration
    - Any text-based data entry
  </Accordion>
  
  <Accordion title="When Not to Use" icon="xmark">
    - For selecting from predefined options (use Select)
    - For binary choices (use Checkbox or Toggle)
    - For date selection (use Date Picker)
    - For large amounts of formatted text (use Rich Text Editor)
  </Accordion>
</AccordionGroup>

### Do's and Don'ts

<CardGroup cols={2}>
  <Card title="Do" icon="check" color="#16A34A">
    - Use clear, descriptive labels for every input
    - Provide helpful validation messages
    - Use appropriate placeholder text as hints
    - Associate labels with inputs using htmlFor/id
    - Provide actionable error messages
  </Card>
  
  <Card title="Don't" icon="xmark" color="#DC2626">
    - Don't replace labels with placeholders
    - Don't use vague labels like "Input" or "Field 1"
    - Don't hide validation messages
    - Don't use vague errors like "Invalid input"
    - Don't overload inputs with unnecessary elements
  </Card>
</CardGroup>

<Tip>Always place labels above inputs for better accessibility and mobile experience.</Tip>

<Warning>Never use placeholder text as a replacement for labels - it disappears when users start typing.</Warning>

## Accessibility

<Info>All input components meet WCAG 2.1 Level AA accessibility standards.</Info>

### Semantic HTML and Labels

<CodeGroup>
```jsx Proper Label Association
<label htmlFor="email">Email Address</label>
<input 
  id="email" 
  type="email" 
  name="email"
  aria-describedby="email-helper"
/>
<span id="email-helper">We'll never share your email</span>
```

```jsx Required Field
<label htmlFor="username">
  Username <span aria-label="required">*</span>
</label>
<input 
  id="username"
  type="text"
  required
  aria-required="true"
/>
```

```jsx Error State
<label htmlFor="password">Password</label>
<input 
  id="password"
  type="password"
  aria-invalid="true"
  aria-describedby="password-error"
/>
<span id="password-error" role="alert">
  Password must be at least 8 characters
</span>
```
</CodeGroup>

## Engineering Implementation

### Component API

```typescript
interface InputProps {
  // Input type
  type?: 'text' | 'email' | 'password' | 'number' | 'search' | 'tel' | 'url';
  
  // Labels and text
  label?: string;
  placeholder?: string;
  helperText?: string;
  errorMessage?: string;
  
  // Appearance
  size?: 'sm' | 'md' | 'lg';
  
  // States
  disabled?: boolean;
  readOnly?: boolean;
  required?: boolean;
  
  // Value management
  value?: string;
  defaultValue?: string;
  onChange?: (event: ChangeEvent<HTMLInputElement>) => void;
  onBlur?: (event: FocusEvent<HTMLInputElement>) => void;
  onFocus?: (event: FocusEvent<HTMLInputElement>) => void;
  
  // Enhancements
  prefix?: ReactNode;
  suffix?: ReactNode;
  clearable?: boolean;
  
  // Validation
  maxLength?: number;
  pattern?: string;
  
  // Accessibility
  autoComplete?: string;
  autoFocus?: boolean;
  ariaLabel?: string;
  ariaDescribedBy?: string;
  
  // Styling
  className?: string;
}

interface TextareaProps extends Omit<InputProps, 'type'> {
  rows?: number;
  cols?: number;
  resize?: 'none' | 'vertical' | 'horizontal' | 'both';
}
```

### Implementation Notes

**Technical Requirements:**
- Use native `<input>` element for proper semantics
- Implement controlled or uncontrolled patterns
- Handle password visibility toggle securely
- Debounce validation for better UX
- Preserve cursor position during controlled updates
- Support autofill styling
- Implement character counter for maxLength
- Handle IME composition events properly

**Form Integration:**
- Associate labels with inputs using htmlFor/id
- Use appropriate autocomplete attributes
- Validate on blur, not on every keystroke
- Provide clear, actionable error messages
- Support form submission with Enter key
